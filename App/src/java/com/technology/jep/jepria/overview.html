Описание

Основная идея платформы JepRia - предоставить пользователю Internet-приложений весь богатый арсенал возможностей пользовательского интерфейса stand-alone -приложений, сделать интерфейс более "живым"/интерактивным/удобным в использовании.
При этом, платформа продолжает следовать основным идеям платформы Jep (JEP/Module/JepCommon) - абсолютно единообразное создание простых интерфейсов по отображению и редактированию справочников - предоставляя качественно более развитый пользовательский интерфейс.

Переход на новые версии GWT

В основу концепции данной библиотеки заложена идея возможности перехода на более новые версии фреймворка Google Web Toolkit.
Однако есть ряд нюансов, которые следует учитывать при переходе на более позднюю версию фреймворка. Поэтому следует:
  • Обновлять файлы gwt-user.jar и gwt-servlet.jar откомпилированными версиями классов Date_CustomFieldSerializer из пакетов {@link com.google.gwt.user.client.rpc.core.java.sql} и {@link com.google.gwt.user.client.rpc.core.java.util} (важно: java-компилятор должен иметь версию 1.6.*)
  • Скопировать файлы validation-api.jar и validation-api-sources.jar из текущей версии GWT в корневую папку GWT обновленной версии
  • В случае необходимости, произвести рефакторинг в коде в связи с соответствующими изменениями в API библиотеки GWT
  • Произвести пересборку JepRia, не забыв о переключении параметра GWT_HOME в build.properties, и убедиться в отсутствии ошибок
Помимо пересборки JepRia, возможно понадобится и переход на обновленную версию IDE.
Для Eclipse также важно произвести следующие действия:
  • Приоставлена поддержка OC4J корпорацией Oracle и данная инициатива была поддержана консорциумом разработки Eclipse (в новых версиях нет поддержки сервер-адаптера для OC4J).

  • Для работы с SVN необходимо установить актуальную версию плагина для работы с SVN - Subclipse (http://subclipse.tigris.org/update_1.10.x/) или любой другой плагин на усмотрение разработчика.
  • Из-под IDE не удавалось соединиться с удаленным Marketplace. Для решения этой проблемы в конфигурационном файле eclipse.ini следует добавить аргумент для запуска среды:

Корректное использование метода {@link java.lang.Object#equals(java.lang.Object)}

При разработке приложений на языке Java следует учитывать некоторые особенности платформы:
  • Всем известно, что при сравнении объектов следует использовать метод {@link java.lang.Object#equals(java.lang.Object)}. Но мало, кто знает, что в нативной реализации данного метода объекты сравниваются по ссылке в памяти. Обычно этого достаточно для кастомных классов, поля которых имеют примитвный тип. Если же в разрабатываемом классе в качестве полей используются объекты более сложных типов, то для сравнения его объектов следует переопределять не только метод {@link java.lang.Object#equals(java.lang.Object)}, но и ОБЯЗАТЕЛЬНО метод {@link java.lang.Object#hashCode()}. При реализации данных методов необходимо придерживаться следующей логики:
    • сперва вычисляются хэш-коды сравниваемых объектов
    • если их значения отличаются, то это явный признак, что объекты различны
    • в противном случае - будет отрабатываться логика метода {@link java.lang.Object#equals(java.lang.Object)}
    Для более точного сравнения объектов следует обеспечивать уникальность хэш-значений объектов.
    Для этого в алгоритме метода {@link java.lang.Object#hashCode()} в качестве основания для вычисления обычно используются простые числа (17, 31 и т.д.), а в качестве зависимых значений выступают хэш-коды полей класса.
    Более того, метод {@link java.lang.Object#hashCode()} используется при извлечении значений из хэш-коллекций (например, {@link java.util.HashMap}), в которых в качестве КЛЮЧА используются объекты данного класса. Уникальность хэш-кодов объекта обеспечивает эффективную работу хэш-коллекции. В противном случае, при поиске ключа будет осуществлен полный перебор коллекции, что значительно повышает временные затраты.
    В случае возникновения сложностей при реализации методов сравнения объектов следует обратиться к имплементациям {@link com.technology.jep.jepria.shared.load.FindConfig#equals(java.lang.Object)} и {@link com.technology.jep.jepria.shared.load.FindConfig#hashCode()} в качестве примера.
  • Однако, если в качестве сравниваемых объектов выступают массивы, то для их корректного сравнения следует использовать статичный метод {@link java.util.Arrays#equals(java.lang.Object[], java.lang.Object[])}. При таком подходе, сравнение массивов осуществляется поэлементно. Причем, если в качестве элементов массива используются объекты кастомных классов, то все рассуждения из предыдущего пункта также верны.

Соглашение об использовании Z-индексов

После релиза JepRia 9.0.0 изменилась политика установления Z-индексов (глубина слоя) графических элементов.
  • Z-индексы любых элементов, которые указываются явно, должны быть написаны в JepRia.css, а не в java-коде. Это необходимо для того, чтобы можно было быстро посмотреть всю иерархию слоёв в приложении.

  • Графические элементы, которые пользователь видит на экране в JepRia-приложениях, были разделены на классы, в зависимости от глубины слоя (z-индекса), на котором этот элемент может находиться (от нижних к верхним):
    • Основной слой: форма, с которой работает пользователь, графические элементы (виджеты), находящиеся непосредственно на форме.
    • Слой маски: слой, на котором находится маска, в то время как на экране появляется диалоговое окно или индикатор загрузки данных (например, из БД).
    • Слой диалогового окна: диалоговые окна, графические элементы (виджеты), находящиеся непосредственно на диалоговом окне, также меню настройки столбцов на списочной форме.
    • Слой всплывающих элементов (Popup): всплывающие элементы виджетов, такие как datepicker, imagepicker, окошко выбора цвета, выпадающий список комбо-бокса.
    • Слой завешивания: слой, на уровне которого завешивается всё окно приложения (с индикатором загрузки). Например: при начальной загрузке модуля, при загрузке файлов на сервер.
    • Слой тестовой сборки: индикатор тестовой сборки в правом верхнем углу экрана.

  • Каждому классу (кроме Основного слоя - там z-индексы явно не присваиваются) соответствует определнный интервал z-индексов, которые может принимать элемент данного класса. Необходимо избегать явного присвоения элементу z-индекса другого класса, поскольку это приводит к перекрытию слоёв и, как следствие, неправильному отрображению виджетов. Классы разрежены пустыми интервалами, для возможного появления новых классов в будущем.
